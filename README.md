1.2.1项目架构

1）定位器
主控芯片：STM32F103C8T6（8M，功耗考虑）。
LoRa模块：E220-400M22S，内置LoRa芯片：LLCC68。

2）接收器
主控芯片：STM32F103C8T6（8M，功耗考虑）。
LoRa模块：E220-400M22S，内置LoRa芯片：LLCC68。
蓝牙模块：ECB01。
2.2.2项目用的哪个单片机？RAM多大，最高主频多少？
用的STM32F103C8T6，RAM是20KB，最高主频72M。
考虑到功能需求和成本，所以选择这款芯片。
接收器中的MCU时钟频率设置8M。

（1）定位器

1.2.4你们用到的4G和GPS模块是如何和MCU进行通信的?

是通过串口，发送AT指令。

1.2.5对于得到的GPS信号是如何发出去的?

（1）通过4G发送到云服务器
（2）通过LoRa发送到接收器

1.2.6对于LoRa通信，据我所知，即使同步字不同也有可能收到对方的数据，你们怎么解决这个问题的?

你说的问题确实存在，我们也碰到了。我们解决方案是，给数据内容定义自己的格式，接收时,先检测格式是否正确，如果格式对，再进行数据处理.如果格式不对，数据直接丢失。
1.2.7既然已经通过4G通信传输GPS信号到了云服务器，为什么还要再有LoRa通信呢，这么设计的依据是是什么?

主要考虑到运营商的4G覆盖还不能做到全覆盖，如果碰到没有4G信号的地方，就没有办法收到定位信息了。
而LoRa是我们自己组网,就不存在这个问题。这个算是一个冗余设计，可以保证在任何情况下都可以接收到定位数据。
1.2.8你们的接收器是如何与手机APP进行连接的?

是靠的蓝牙通信。手机主动通过蓝牙连接接收器的蓝牙，然后手机端就可以收到定位数据,然后进行可视化展示。
我们公司有专门的客户端工程师来开发APP。

1.2.9定位器接收的经纬度信息那些数据里面除了经纬度还有什么？


旧版（飞思创）：0,0,0,0,0 ==》经度，纬度，对地速度，海拔，时间
新版：NMEA 0183格式的GPS信息。这些信息包含了当前位置、时间、卫星状态等信息。只要 $GNRMC 的数据：
$GNRMC,075504.000,A,4006.24628,N,11621.60497,E,0.00,0.00,140624,,,A,V*0D
$GNRMC：消息类型
075504.000：UTC时间07:55:04.000
A：状态，A=有效定位
4006.24628,N：纬度40度6.24628分北纬
11621.60497,E：经度116度21.60497分东经
0.00：地面速率（节）
0.00：地面航向（度）
140624：日期14-06-2024
,,：磁偏角信息（空字段）
A：定位模式，A=自主定位
V：警告状态，V=无效
0D：校验和
完整数据如下（仅了解）：
    $GNGGA,075504.000,4006.24628,N,11621.60497,E,1,07,1.3,110.5,M,-9.6,M,,*52
    $GNGLL,4006.24628,N,11621.60497,E,075504.000,A,A*45
    $GNGSA,A,3,05,18,20,29,,,,,,,,,2.6,1.3,2.3,1*33
    $GNGSA,A,3,16,19,34,,,,,,,,,,2.6,1.3,2.3,4*3B
    $GPGSV,2,1,08,05,69,010,33,07,11,045,,13,69,151,,15,47,216,,0*69
    $GPGSV,2,2,08,18,20,310,30,20,52,062,25,29,48,265,27,30,23,075,,0*62
    $BDGSV,1,1,03,16,12,172,30,19,48,046,31,34,11,320,31,0*44
    $GNRMC,075504.000,A,4006.24628,N,11621.60497,E,0.00,0.00,140624,,,A,V*0D
    $GNVTG,0.00,T,,M,0.00,N,0.00,K,A*23
    $GNZDA,075504.000,14,06,2024,00,00*4C
    $GPTXT,01,01,01,ANTENNA OPEN*25
数据说明：
$GNGGA：全球定位系统修正数据
$GNGLL：地理位置数据
$GNGSA：GPS DOP和活动卫星
$GPGSV：可见卫星信息
$BDGSV：北斗卫星系统可见卫星信息

1.2.10项目最终串讲

这个定位器项目，当时公司出了几款针对不同场景的型号。这边我负责的是针对宠物定位这个场景的型号。宠物定位器，允许宠物主人实时追踪和监控宠物的位置。（其他型号有针对大规模养殖畜牧业的产品、针对老人小孩定位的产品，其他同事负责的）
项目中会涉及三个设备：宠物定位器、定位接收器、手机（App展示）。其中手机端的App是负责软件开发的同事负责的，我们这边就是将数据上报给手机，以便App做一个直观展示。定位器会使用到飞思创的GPS+4G模块、LoRa模块，接收器有Lora模块、国产的蓝牙模块ECB01。定位器和接收器使用的MCU是STM32F103C8T6，主频最高72M，RAM20K，考虑到功耗我们在使用的时候主频都设置为8M。
整个项目我们是做了一个分层的规划，目的是提高系统的可维护性、可扩展性和可重用性。分层包括：
驱动层：主要是片内外设的驱动，与芯片直接交互的代码；
硬件接口层：主要是外部硬件的驱动，提供统一接口给上层应用；
公共层：主要是一些工具函数；
应用层：主要的业务逻辑；
接下来介绍一下项目的实现细节：
定位器的GPS模块接收到定位信息之后，通过串口发送到主控芯片，之后通过无线通信最终上报给手机App，这里上报的方式有2种。
第一种方式是通过定位器的LoRa模块发送到接收器的LoRa模块，因为相比于蓝牙、wifi这些短距离无线通信方式，LoRa可以支持几公里到十几公里的距离，而且这个定位信息还有它数据发送频率没那么高，所以 LoRa 虽然它支持的速率并不是特别高，但也能满足需求。接收器通过 Lora 接收到定位信息之后，通过 SPI 将数据发送到 MCU 上面（可以展开介绍SPI）。之后这个 MCU 将数据通过蓝牙的方式推送到手机端，手机端App这边我们是我们公司专门软件开发去负责开发的一个APP。这个蓝牙我们用的是一个国产的 ECB01 蓝牙模块，图片MCU通过串口与蓝牙通信。这里选择蓝牙主要是考虑到，一是低功耗，续航更久，避免充电。第二是考虑到无线传输距离，是一个近距离的，因为在接收器一般带在身边，然后身上又带着手机，这个距离一点问题没有。另外一个速率要求也不高，那蓝牙也完全顶得住，所以这边我们选择是蓝牙模块，而这个蓝牙是出厂，它本身自带一个官方固件。那我们可能要去修改它的一些配置，可以通过 at 指令的方式去做修改。
第二种方式是在LoRa失去连接时的备用方式。如果宠物离开距离太远，或者接收器断电了时，就无法通过LoRa来接收。当定位器这边发现LoRa连接不上，就会启动4G模块，将定位信息通过4G模块推送到云服务器，再推送到手机App。如果4G模块一直开启，会导致定位器的功耗过高，续航变低，所以在这里4G只是在LoRa失联后才会去主动开启。在启用4G期间，会每隔10分钟判断LoRa是否可以连接，如果恢复连接了就主动关闭4G，继续使用LoRa传输数据。
